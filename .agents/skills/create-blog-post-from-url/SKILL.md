当用户提供一个文章 URL 并要求创建博客时，按下面的简化流程执行。

## 抓取方式

网页内容获取默认使用 **chrome-devtools**，不要只依赖普通 HTTP 抓取。

推荐顺序：

1. 用 chrome-devtools 打开原文页面
2. 等待正文或关键标题出现
3. 读取页面快照，提取标题、正文结构、作者、日期、配图、代码块
4. 如果正文需要展开（例如"Read more""展开全文""Show more"），先交互再读取
5. 如果文章包含强相关延伸链接，再继续打开并提取这些页面

只有在 chrome-devtools 明显不可用，或者页面内容无需浏览器渲染时，才把普通网页抓取作为补充手段。

### chrome-devtools 不可用时的 fallback

当 chrome-devtools 不在当前 agent 工具集中时，按以下顺序回退：

1. **curl 下载 HTML**：`curl -sL --max-time 30 "<URL>"` 保存原始 HTML
2. **提取结构化元数据**：从 `<title>`、`<meta name="description">`、`<meta property="og:*">`、`<meta name="author">`、`<meta property="article:published_time">` 中提取关键信息
3. **提取正文**：移除 `<script>`、`<style>`、`<nav>`、`<footer>`、`<header>` 标签后，从 `<article>` 或 `<main>` 或 `<body>` 中剥离 HTML 标签获取纯文本
4. **结构化笔记**：将提取到的内容按章节标题分组为结构化笔记，而不是整页原文机械复制

**限制**：此方式无法处理 JS 渲染的 SPA 页面、懒加载内容、折叠区域或需要交互展开的正文。如果页面依赖 JS 渲染，优先告知用户这一限制并建议手动提供关键内容。

### 超长内容处理

当原文超过 20 个独立要点/章节时（例如长篇 Q&A、系列教程），不要机械地逐条翻译。

1. **先读取目录/章节结构**，理清文章分类框架
2. **每类保留 2-3 个代表性题目/步骤**做详细展开
3. **其余内容合并为要点列表**，用简洁的总结句带过
4. **文末注明**"完整内容请参考原文链接"

判断标准：最终博客正文控制在 8000-15000 字以内比较合理。超过这个范围，读者在信息流中的完成阅读率会显著下降。

## 核心目标

写出一篇**中文、可信、可读、可落地**的博客文章。

始终记住这 5 条：

1. **说人话**：像熟悉技术的一线同行在解释问题，不要像宣发文案。
2. **先讲价值**：读者读完要知道这东西能帮他做什么。
3. **别制造焦虑**：不要写“再不学就晚了”“即将被淘汰”这类话。
4. **基于原文**：保留原文的具体信息，不借题发挥。
5. **教程要能照着做**：如果是实操型内容，步骤必须清楚、细致、可验证。

## 输入

- 必需：原文 URL
- 可选：标题、原文发布日期、标签
- 可选：是否生成封面、是否发布到微信公众号草稿箱

`原文发布日期` 只用于理解上下文，不用于 frontmatter 里的 `pubDatetime`。

## 1. 先收集必要信息

至少提取这些内容：

- 标题
- 作者 / 来源
- 原文发布日期
- 核心结论或主题
- 章节结构
- 关键截图、图表、流程图或代码片段

### 用 chrome-devtools 提取时的要求

- 优先读取页面标题、主标题、正文容器、目录、代码块、图片说明、作者信息、发布时间
- 如果页面是单页应用或内容延迟加载，先等待关键文本出现，再读取快照
- 如果页面弹出 Cookie、登录提示、订阅浮层等遮挡正文，先尝试关闭，再继续读取
- 如果正文分多页、Tab 或折叠区域，优先补齐和主题直接相关的部分
- 提取到的内容要转成结构化笔记，避免把整页原文机械复制进博客

只保留和主题直接相关的信息，不整段照搬原文。

如果原文包含延伸链接、Part 2、官方文档或示例仓库，只跟进和正文强相关的链接。

### 跟进相关链接的规则

- 只跟进正文中明确引用、而且会影响理解或操作步骤的链接
- 优先级通常是：Part 2 / 官方文档 / 示例仓库 / 文章里直接引用的数据来源
- 每跟进一个链接，都要记录它补充了什么信息，避免读完很多页面却没有进入正文
- 如果链接只是导航、营销页、作者主页、社交媒体讨论，通常跳过

## 2. 先做公众号运营判断

这一步用于决定文章是否只是博客归档，还是也适合微信公众号分发。默认账号定位按 **Aide Hub** 处理：

- AI 助手
- 软件开发
- 技术洞察

动笔前先判断：

1. 这篇文章服务哪个账号定位？如果都不匹配，仍可写博客，但不要强行包装成公众号爆款。
2. 读者读完能获得什么：学会一个方法、形成一个判断、拿到一套工具，还是避开一个坑？
3. 内容价值类型是否足够明确：优先写“有用”的内容，不写空泛趋势、情绪煽动或焦虑话术。
4. 这篇文章在微信里更适合被收藏、转发还是关注？后面的标题、摘要、封面和文末引导都要服从这个判断。

常见公众号分发倾向：

- **教程 / 实操型**：更适合收藏，重点写步骤、结果验证、坑点
- **观点评论型**：更适合转发，重点写判断依据、适用边界
- **工具评测型**：更适合涨粉，重点写使用场景、替代方案、上手门槛
- **产品更新 / 发布型**：更适合帮助读者判断是否值得关注，不要写成厂商通稿
- **研究 / 论文解读型**：更适合建立信任，重点讲清核心发现、限制和现实意义

不要把每篇外部文章都硬改成“爆款公众号文”。如果原文只适合博客沉淀，就保持技术博客写法，只在标题和摘要里说明清楚价值。

## 3. 先判断文章类型，再决定写法

常见类型：

- **教程 / 实操型**：重点写清楚问题、前提、步骤、结果、坑点
- **产品更新 / 发布型**：重点写更新了什么、适合谁、值不值得关注
- **观点评论型**：重点写原作者观点、判断依据、适用边界
- **研究 / 论文解读型**：重点写核心发现、限制、现实意义
- **案例复盘型**：重点写背景、决策、过程、教训
- **工具评测型**：重点写体验、优缺点、适用场景、替代方案

动笔前先回答：

1. 这篇文章最重要的是“教会别人做事”还是“帮助别人形成判断”？
2. 原文最有价值的是方法、观点，还是新信息？
3. 读者读完应该带走哪一个结果？

后面的结构、段落、图片和结尾，都服从这个判断。

如果原文页面信息不完整，要先判断是因为：

- 页面尚未加载完成
- 正文被交互元素遮挡
- 内容在 iframe、代码折叠块或懒加载区域中
- 文章核心信息其实分散在原文和它引用的官方资料里

先补齐信息，再写，不要在信息残缺的情况下硬生成成文。

## 4. 写作要求

### 随机风格选择

- 每篇文章在动笔前先确定 `selected_style`：如果用户明确指定风格，使用用户指定；否则从 `references/cover-prompt-template.md` 的 40 项风格池中随机选择 1 项
- `selected_style` 一篇文章只选一次，文章表达、封面 brief、封面 prompt、图片验收都使用同一个风格方向
- 风格只影响叙事语气、视觉隐喻、配色、构图和图片 prompt；不得改变原文事实、技术判断、步骤、数字或引用关系
- 文章正文仍保持中文技术博客的自然、可信、可落地表达，不要为了套风格写成小说、角色对白或营销文案
- 用户明确说不要封面、不要生成图片或只保留正文图片时，跳过图片生成；但仍可在写作时记录并轻量遵循同一个 `selected_style`
- 对强指向品牌、工作室、作者或特定在世艺术家的风格名称，只把原名记录在 brief 的 `selected_style`，生成 prompt 时必须使用 `selected_style_prompt_descriptor` 的非仿冒描述词，不要求复刻某个品牌、工作室或艺术家的画风

### 基础要求

- 全文必须是中文，标题、摘要、正文、图片说明都要翻译
- 专有名词保留英文，首次出现时可补中文说明
- 至少保留 **2-4 个原文锚点**，比如原作者的分类、例子、数据点、实验现象、判断句

### 风格要求

- 用朴素、直接、自然的表达
- 开头两段内就让读者知道：这篇能帮他理解什么，或者做成什么
- 先解释动机，再给代码、配置或步骤
- 结尾自然收住，可以写适用边界、实践建议或后续观察点

### 禁止项

- 不要用“首先、其次、最后”“总之”“值得注意的是”这类明显 AI 腔
- 不要用“不是 X，而是 Y”“真正重要的是”这类高频套路句
- 不要用夸张、焦虑、空洞黑话
- 不要把观点评论型文章硬改成教程

## 5. 教程型文章要重点写清楚步骤

如果原文属于**教程 / 实操型**，必须把“能不能照着做出来”放在第一位。

### 教程正文至少要交代清楚这些内容

1. **要解决什么问题**
	- 读者为什么要做这件事
	- 适合什么场景，不适合什么场景

2. **前置条件**
	- 需要的软件、版本、账号、权限、环境变量
	- 有哪些默认前提不能省略

3. **操作步骤**
	- 按顺序写，不能跳步
	- 每一步都写清楚“做什么、为什么这样做、做完会看到什么结果”
	- 如果一步里有关键输入、命令、按钮、路径、配置项，要写完整

4. **结果验证**
	- 告诉读者怎样确认这一步是成功的
	- 有报错、输出、界面变化、文件变化时，要明确写出来

5. **常见坑和注意事项**
	- 哪些地方最容易做错
	- 出错时优先检查什么

### 教程场景下的页面取证要求

- 命令、配置、按钮名称、路径、错误提示，优先从页面原文或官方链接中逐项核对
- 如果原文是视频页、演示页或长线程摘要，必须把真正可执行的步骤补齐后再写
- 页面截图只能作为辅助线索，真正写进步骤里的操作要以正文、代码块、文档说明为准

### 教程写法规则

- 步骤多时，用 `##` / `###` 分阶段，不要堆成一大段
- 单个步骤内部可以用短列表，但正文仍要有解释，不要只剩命令清单
- 代码和命令必须尽量完整，避免只贴关键几行让读者自己猜
- 出现截图时，优先保留步骤图、界面图、结果图
- 如果原文步骤本身不够细，要在不脱离原意的前提下补足必要说明

判断标准很简单：**一个熟悉该领域、但没做过这件事的读者，能不能顺着文章做下来。**

## 6. 标题、摘要和小标题

### 主标题

- 先准确概括主题，再考虑是否加入轻微判断
- 降低解释成本，让读者一眼看出文章对象和读完能得到什么
- 保留技术名词、产品名、场景词
- 避免标题党、悬念句、过度包装
- 避免生僻词、抽象大词、过度聪明但难传播的表达
- 优先使用“技术对象 + 具体收益 / 适用场景”的结构，例如“Claude Code Hooks：把重复检查自动化”这类直观标题

### 摘要

- frontmatter `description` 同时服务博客列表和微信预览，控制在 80-120 字
- 开头直接说明这篇文章解决什么问题、适合谁、读完能带走什么
- 不写"本文深入探讨""重磅发布""颠覆行业"等空泛或焦虑表达
- 不承诺原文没有提供、账号也没有实际准备的资料、源码、课程或福利
- **禁止在 description 中使用中文引号 `""`、`''` 或其他非 ASCII 引号字符**；中文引号在 YAML 双引号字符串内会导致解析异常和渲染问题。如原文需要用引号强调，改用「」或去掉引号改用自然表达

### 小标题

- 尽量短，最好 6-12 个中文字符，超过 20 个就改短
- 小标题应该像目录，不像正文第一句
- 避免连续多个标题用同样的起手词

## 7. 生成 ID 与 Slug

- 读取 `src/data/blog` 现有文件，解析数字前缀并递增，保持 3 位
- `slug` 使用 kebab-case，保持唯一

## 8. 图片处理

图片保存到 `src/assets/{ID}/`。

### 保留原则

- 优先保留和正文强相关的图片：步骤截图、结果图、流程图、架构图、图表
- 过滤导航图标、头像、广告图、无信息量的小图
- 教程文章优先保留步骤截图，观点文章宁缺毋滥

### 封面图

- 默认生成一张文章封面，并把最终选定文件保存到 `src/assets/{ID}/01-cover.{ext}`
- 如果用户明确说不要封面、不要生成图片或只保留正文图片，跳过封面生成，不写 `ogImage`
- 封面生成流程默认**不指定某个固定图像生成 backend**；重点是先完成封面 brief、产出合格草图、再把最终选中图落盘到 `src/assets/{ID}/01-cover.{ext}`
- 如果所选图像技能先输出到临时目录、默认目录或中间文件，最终都要移动或复制到 `src/assets/{ID}/01-cover.{ext}`；不得让 `ogImage`、正文或微信封面引用仍指向临时生成路径
- 如果用户明确要求 Azure / `azure-image-gen` / Azure OpenAI，必须尊重该要求；此时以后处理后的**最终实际路径**和扩展名为准（可能是 `.png`、`.jpg` 或 `.webp`）
- 无论使用哪个后端，都要同步更新正文引用、`ogImage` 和微信封面路径
- 封面用于表达主题，不用于伪造证据
- 不得生成会让人误以为是真实产品截图、数据图或架构图的图片
- 生成封面前，先读取 `references/cover-prompt-template.md`
- `create-blog-post-from-url` 的默认封面，不再是“泛主题插画”，而是**用单张宽封面讲清文章主线的 infographic storytelling summary**：在一张 `2.35:1` 宽图里，用 `1` 个主焦点 + `2-4` 个叙事信息块，概括文章结论、机制、步骤或对比
- 生成 brief 前，先从原文抽出：`1` 个主结论、`2-4` 个可视化信息块（步骤 / 证据 / 转折 / 对比）、以及它们各自对应的原文锚点；封面上的信息必须来自这些锚点，不能临场编造数字、标签或节点名
- 宽封面里最多放 `2-4` 个信息块；超过这个数量，先压缩叙事，不要把封面做成难以辨认的小字长海报
- 先判断封面类型，再按模板填写提示词槽位，保存 `src/assets/{ID}/cover-brief.json`，不要临场随手拼一段泛化描述
- `cover-brief.json` 除了现有主题、主体、构图、风格字段外，还要记录 `summary_angle`、`narrative_flow`、`information_blocks`、`selected_style`、`selected_style_prompt_descriptor`、`selected_style_reason` 与 `style_safety_note`
- 封面的最终视觉风格使用写作前选定的 `selected_style`；不要退回普通 editorial 插画、产品海报、抽象科技氛围图，也不要改成另一个随机风格
- `selected_style_prompt_descriptor` 必须来自 `references/cover-prompt-template.md` 的安全描述映射；如果用户指定了不在清单内的风格，也要先转成不仿冒特定品牌、工作室或艺术家的描述词
- 每次随机或用户指定风格都要把选择原因写进 `selected_style_reason`，并在 `style_safety_note` 里说明是否做了安全改写
- 允许少量可见文字，但只能是**来源可追溯的中文短标签 / 数字**：默认 `1-3` 处，确有必要时最多 `4` 处；每处不超过 `6` 个汉字，或 `1` 个必要的产品 / 模型名；允许序号 `1/2/3`、`①②③`、原文真实数字或百分比
- 仍然禁止：封面主标题、长句说明、英文口号、假 UI 标签、假 dashboard、假图表坐标 / 图例、假终端输出、假架构节点名、伪造指标
- 默认使用**封面 brief -> 草图 -> 验收 -> 精修/重生 -> 落盘**流程，而不是一次性生成后直接收工：
  1. **封面 brief**：记录文章类型、核心主题、读者收获、主隐喻、主视觉主体、主体动作、构图、背景元素、基础风格、选定随机风格、安全风格描述、色彩、叙事摘要、信息块、禁用元素、参考素材策略和最终 prompt
  2. **草图阶段**：先用纯文本 prompt 验证“主题、主隐喻、叙事信息块、选定风格、构图”是否成立
  3. **精修阶段**：如果第一张已经接近目标，但细节、风格、背景、误生成元素还有问题，优先编辑或定向重生 1-2 轮；每轮只改 1-3 个明确问题
  4. **落盘阶段**：只把最终选定图作为 `01-cover.{ext}` 写入文章资源目录，丢弃的预览图不进入正文引用
- 如果文章已经保留了**允许复用的本地素材**（例如你刚保存到当前文章资源目录中的步骤图、结构草图、结果图），而且这些素材确实能帮助模型抓住主题，可以把其中 1-2 张作为参考输入交给生图工具
- 但不要把第三方版权海报、品牌 KV、原站营销插画或需要精确复刻的产品界面直接拿去做“像素级临摹”；参考图只用于帮助模型理解主体关系、空间结构和关键对象，不用于复制原图
- 封面默认是宽图，目标比例为 2.35:1；如果工具不能精确指定尺寸，也要在 prompt 中明确 `wide cover illustration, aspect ratio 2.35:1`
- 不要依赖图中长文字表达主题；封面标题仍由博客页面负责，图中只允许少量中文短标签 / 数字承担 infographic 导航作用，不要混入整句英文
- 如果生成结果把画面带往“海报字样 / UI 截图 / dashboard / 假终端 / 假图表”，立刻收紧 prompt 后重生或编辑，不要接受
- 如果用户明确要求 `azure-image-gen`，再使用它的参数规则：草图优先 `1504x640` / `quality low`，终稿优先 `2256x960` / `quality medium`，`background` 只用 `auto` 或 `opaque`

### 封面图验收标准

- 一眼能看出文章主题，而且能看出**至少 2 个来自文章本身的叙事节点**，而不是“泛 AI 科技图”
- 只有一个主焦点；其余信息块用于讲清主线，不抢戏，也不堆成密集小字海报
- 可见文字只允许 `1-3` 处少量中文短标签 / 数字，并且必须能映射回 `cover-brief.json` 里的 `information_blocks`；不得出现主标题、长句、英文口号
- 没有水印、假界面、假 dashboard、假图表坐标 / 图例、假终端、假架构标签、伪造指标
- 如果用了参考图或编辑流，最终画面也不能看起来像“原网页截图换了滤镜”
- 如果第一张图主题对了但摘要不够清楚或信息块不对，先回到 brief 压缩叙事，再走编辑流；如果主题本身就错了，回到 prompt 槽位重新设计，不要硬修

### 网页图片提取要求

- 用浏览器上下文判断图片是否和正文强相关，不要只看文件名
- 优先保留正文里的步骤图、结果图、架构图、图表
- 如果图片点开后才是高清版本，可以进一步打开或下载对应资源
- 对于纯装饰性插图、头像、按钮图标、社交分享图，通常不保留

### Markdown 规则

- 本地图片路径写为 `../../assets/{ID}/{filename}`
- 图片尽量放在对应内容附近，不要全部堆到文末
- 每张图补中文 `alt`
- 若需要分享图，`ogImage` 必须指向**封面最终实际文件**，例如 `../../assets/{ID}/01-cover.png`、`../../assets/{ID}/01-cover.jpg` 或 `../../assets/{ID}/01-cover.webp`

## 9. 生成 Markdown 文件

文件路径：`src/data/blog/{ID}-{slug}.md`

### frontmatter 要求

- `pubDatetime` 不复用原文发布日期，默认使用**安全发布时间**：
  - 以系统当前时间为基准，转换为 `Asia/Shanghai`（`+08:00`）
  - 再减去 `20` 分钟后写入 frontmatter（避免发布后因未来时间被列表过滤）
- `pubDatetime` 必须是 ISO 8601 且带时区偏移（例如 `+08:00`）
- 禁止使用无时区格式（如 `2026-03-23T10:00:00`）和纯日期格式（如 `2026-03-23`）
- 仅当用户明确要求"定时发布"时才允许未来时间；否则一律使用安全发布时间
- **`description` 禁止出现中文双引号 `""` 和中文单引号 `''`**：这些字符在 YAML 双引号字符串内会导致解析错误。需要引用或强调时改用直角引号「」或直接去掉引号，用自然语言表达
- `title`、`description` 等 YAML 字符串值中同样避免出现 `""` `''`，统一使用「」替代

```yaml
---
pubDatetime: YYYY-MM-DDTHH:mm:ss+08:00
title: "中文标题"
description: "80-120 字中文摘要"
tags: ["Tag1", "Tag2"]
slug: "{slug}"
# ogImage: "../../assets/{ID}/01-cover.{ext}"
source: "{original_url}"
---
```

## 10. 微信公众号文末模块

如果文章会发布到微信公众号草稿箱，默认在 `## 参考` 之前加入自然的文末引导；如果用户明确说只生成博客、不发微信，可以省略这个模块。

文末模块要求：

- 用一小段自然文字说明账号会持续分享什么，默认围绕 AI 助手、软件开发、技术洞察
- 语气克制，不写“赶紧关注”“错过后悔”“全网最全”等营销腔
- 如果文章适合引流，可以补一句“关注后你会持续看到哪类内容”，但不要打断正文结尾
- 可选生成自动回复关键词建议，例如资料、清单、源码、模板；只有在文章确实产出了对应资源、或用户明确提供资源时才写进正文
- 不承诺不存在的资料、课程、社群、源码、优惠、咨询服务或账号能力

推荐结构：

```markdown
如果你关注 AI 助手、开发工具和软件工程实践，可以关注 Aide Hub。这里会继续分享能落地的工具教程、技术观察和项目经验。
```

如果需要关键词建议，在最终报告里单独提示即可，不要把未配置的自动回复当作已生效能力写进文章。

## 11. 文末参考链接

保留一个 `## 参考` 小节，至少包含原文链接。

只保留正文中真正引用过、确实有价值的外部链接，控制在 10 条以内。

## 12. 最终检查

发布前逐项确认：

- [ ] 全文是自然中文，不像模板化 AI 输出
- [ ] 保留了原文的关键锚点，没有借题发挥
- [ ] 已完成公众号运营判断：账号定位、读者收益、分发倾向都清楚
- [ ] 标题准确、解释成本低，小标题够短、够清楚
- [ ] 摘要适合作为微信预览，不夸张、不承诺不存在的资源
- [ ] 如果是教程型文章，步骤已经写清楚：前提、操作、结果验证、坑点都交代了
- [ ] 如果会发布到微信，文末关注引导自然克制，并且没有把未配置的关键词回复写成已生效能力
- [ ] 图片与正文对应，路径正确，`alt` 有信息量
- [ ] `pubDatetime` 是带时区的安全发布时间（默认当前时间减 20 分钟），不是原文日期
- [ ] `description`、`title` 中不含中文引号 `""` `''`（已改用「」或自然表达）
- [ ] `tags`、`slug`、`source` 填写完整
- [ ] 通读一遍，没有空话、套话、焦虑话术
- [ ] 原文内容已优先通过 chrome-devtools 核对，必要的强相关链接也已补充阅读

## 13. 发布到微信公众号草稿箱

Markdown 文件创建完成并通过最终检查后，默认自动使用 **wechat-draft** 技能将文章发布到微信公众号草稿箱。若用户明确说“不发布”“不要发微信”“只生成博客”，则跳过发布步骤。

### 发布参数映射

| wechat-draft 参数 | 来源 |
|---|---|
| `title` | frontmatter 中的 `title` |
| `content-file` | 刚创建的 Markdown 文件路径（`src/data/blog/{ID}-{slug}.md`） |
| `content-format` | `markdown` |
| `author` | `Aide Hub` |
| `digest` | frontmatter 中的 `description` |
| `cover-image` | 封面图最终实际路径（例如 `src/assets/{ID}/01-cover.png` / `.jpg` / `.webp`），如果存在 |
| `content-source-url` | frontmatter 中的 `source`（原文 URL） |

### 执行规则

- Look for a `.env` file in the project directory or parent directories
- `app_id` 和 `app_secret` 从 `.env` 文件中的 `WECHAT_APP_ID` / `WECHAT_APP_SECRET` 读取，不要向用户索要
- 发布成功后向用户展示返回的 `media_id`，并提醒去微信公众平台后台检查草稿箱

## 14. Git 提交与推送

如果微信公众号草稿箱发布成功，并且 `wechat-draft` 已返回 `media_id`，继续在博客项目目录执行 git 提交与推送。

如果用户明确说“不发布”“不要发微信”“只生成博客”，跳过微信公众号发布步骤，同时跳过本节 git 流程。

如果微信公众号发布失败，停止 git 提交与推送，向用户报告失败原因和已经生成的本地文件。

### 提交范围

只提交本次文章相关文件，避免带入无关改动：

- `src/data/blog/{ID}-{slug}.md`
- `src/assets/{ID}/` 下的封面、正文图片、`cover-brief.json` 和其他本次生成资源

### 执行规则

1. 运行 `git status --short --branch`，确认当前分支、远端跟踪关系和工作区改动。
2. 只 stage 本次文章相关文件，不要使用会包含无关改动的宽泛路径。
3. 使用 `git commit -m "Add blog post: {title}"` 提交，`{title}` 使用 frontmatter 中的中文标题。
4. 推送当前分支到它的 upstream。
5. 如果当前分支没有 upstream，推送到 `origin` 的同名分支并设置 upstream。
6. 完成后向用户报告 `media_id`、commit hash 和推送分支。
