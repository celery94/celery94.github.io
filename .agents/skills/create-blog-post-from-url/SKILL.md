当用户提供一个文章 URL 并要求创建博客时，按下面的简化流程执行。

## 抓取方式

网页内容获取默认使用 **chrome-devtools**，不要只依赖普通 HTTP 抓取。

推荐顺序：

1. 用 chrome-devtools 打开原文页面
2. 等待正文或关键标题出现
3. 读取页面快照，提取标题、正文结构、作者、日期、配图、代码块
4. 如果正文需要展开（例如“Read more”“展开全文”“Show more”），先交互再读取
5. 如果文章包含强相关延伸链接，再继续打开并提取这些页面

只有在 chrome-devtools 明显不可用，或者页面内容无需浏览器渲染时，才把普通网页抓取作为补充手段。

## 核心目标

写出一篇**中文、可信、可读、可落地**的博客文章。

始终记住这 5 条：

1. **说人话**：像熟悉技术的一线同行在解释问题，不要像宣发文案。
2. **先讲价值**：读者读完要知道这东西能帮他做什么。
3. **别制造焦虑**：不要写“再不学就晚了”“即将被淘汰”这类话。
4. **基于原文**：保留原文的具体信息，不借题发挥。
5. **教程要能照着做**：如果是实操型内容，步骤必须清楚、细致、可验证。

## 输入

- 必需：原文 URL
- 可选：标题、原文发布日期、标签

`原文发布日期` 只用于理解上下文，不用于 frontmatter 里的 `pubDatetime`。

## 1. 先收集必要信息

至少提取这些内容：

- 标题
- 作者 / 来源
- 原文发布日期
- 核心结论或主题
- 章节结构
- 关键截图、图表、流程图或代码片段

### 用 chrome-devtools 提取时的要求

- 优先读取页面标题、主标题、正文容器、目录、代码块、图片说明、作者信息、发布时间
- 如果页面是单页应用或内容延迟加载，先等待关键文本出现，再读取快照
- 如果页面弹出 Cookie、登录提示、订阅浮层等遮挡正文，先尝试关闭，再继续读取
- 如果正文分多页、Tab 或折叠区域，优先补齐和主题直接相关的部分
- 提取到的内容要转成结构化笔记，避免把整页原文机械复制进博客

只保留和主题直接相关的信息，不整段照搬原文。

如果原文包含延伸链接、Part 2、官方文档或示例仓库，只跟进和正文强相关的链接。

### 跟进相关链接的规则

- 只跟进正文中明确引用、而且会影响理解或操作步骤的链接
- 优先级通常是：Part 2 / 官方文档 / 示例仓库 / 文章里直接引用的数据来源
- 每跟进一个链接，都要记录它补充了什么信息，避免读完很多页面却没有进入正文
- 如果链接只是导航、营销页、作者主页、社交媒体讨论，通常跳过

## 2. 先判断文章类型，再决定写法

常见类型：

- **教程 / 实操型**：重点写清楚问题、前提、步骤、结果、坑点
- **产品更新 / 发布型**：重点写更新了什么、适合谁、值不值得关注
- **观点评论型**：重点写原作者观点、判断依据、适用边界
- **研究 / 论文解读型**：重点写核心发现、限制、现实意义
- **案例复盘型**：重点写背景、决策、过程、教训
- **工具评测型**：重点写体验、优缺点、适用场景、替代方案

动笔前先回答：

1. 这篇文章最重要的是“教会别人做事”还是“帮助别人形成判断”？
2. 原文最有价值的是方法、观点，还是新信息？
3. 读者读完应该带走哪一个结果？

后面的结构、段落、图片和结尾，都服从这个判断。

如果原文页面信息不完整，要先判断是因为：

- 页面尚未加载完成
- 正文被交互元素遮挡
- 内容在 iframe、代码折叠块或懒加载区域中
- 文章核心信息其实分散在原文和它引用的官方资料里

先补齐信息，再写，不要在信息残缺的情况下硬生成成文。

## 3. 写作要求

### 基础要求

- 全文必须是中文，标题、摘要、正文、图片说明都要翻译
- 专有名词保留英文，首次出现时可补中文说明
- 至少保留 **2-4 个原文锚点**，比如原作者的分类、例子、数据点、实验现象、判断句

### 风格要求

- 用朴素、直接、自然的表达
- 开头两段内就让读者知道：这篇能帮他理解什么，或者做成什么
- 先解释动机，再给代码、配置或步骤
- 结尾自然收住，可以写适用边界、实践建议或后续观察点

### 禁止项

- 不要用“首先、其次、最后”“总之”“值得注意的是”这类明显 AI 腔
- 不要用“不是 X，而是 Y”“真正重要的是”这类高频套路句
- 不要用夸张、焦虑、空洞黑话
- 不要把观点评论型文章硬改成教程

## 4. 教程型文章要重点写清楚步骤

如果原文属于**教程 / 实操型**，必须把“能不能照着做出来”放在第一位。

### 教程正文至少要交代清楚这些内容

1. **要解决什么问题**
	- 读者为什么要做这件事
	- 适合什么场景，不适合什么场景

2. **前置条件**
	- 需要的软件、版本、账号、权限、环境变量
	- 有哪些默认前提不能省略

3. **操作步骤**
	- 按顺序写，不能跳步
	- 每一步都写清楚“做什么、为什么这样做、做完会看到什么结果”
	- 如果一步里有关键输入、命令、按钮、路径、配置项，要写完整

4. **结果验证**
	- 告诉读者怎样确认这一步是成功的
	- 有报错、输出、界面变化、文件变化时，要明确写出来

5. **常见坑和注意事项**
	- 哪些地方最容易做错
	- 出错时优先检查什么

### 教程场景下的页面取证要求

- 命令、配置、按钮名称、路径、错误提示，优先从页面原文或官方链接中逐项核对
- 如果原文是视频页、演示页或长线程摘要，必须把真正可执行的步骤补齐后再写
- 页面截图只能作为辅助线索，真正写进步骤里的操作要以正文、代码块、文档说明为准

### 教程写法规则

- 步骤多时，用 `##` / `###` 分阶段，不要堆成一大段
- 单个步骤内部可以用短列表，但正文仍要有解释，不要只剩命令清单
- 代码和命令必须尽量完整，避免只贴关键几行让读者自己猜
- 出现截图时，优先保留步骤图、界面图、结果图
- 如果原文步骤本身不够细，要在不脱离原意的前提下补足必要说明

判断标准很简单：**一个熟悉该领域、但没做过这件事的读者，能不能顺着文章做下来。**

## 5. 标题和小标题

### 主标题

- 先准确概括主题，再考虑是否加入轻微判断
- 保留技术名词、产品名、场景词
- 避免标题党、悬念句、过度包装

### 小标题

- 尽量短，最好 6-12 个中文字符，超过 20 个就改短
- 小标题应该像目录，不像正文第一句
- 避免连续多个标题用同样的起手词

## 6. 生成 ID 与 Slug

- 读取 `src/data/blog` 现有文件，解析数字前缀并递增，保持 3 位
- `slug` 使用 kebab-case，保持唯一

## 7. 图片处理

图片保存到 `src/assets/{ID}/`。

### 保留原则

- 优先保留和正文强相关的图片：步骤截图、结果图、流程图、架构图、图表
- 过滤导航图标、头像、广告图、无信息量的小图
- 教程文章优先保留步骤截图，观点文章宁缺毋滥

### 封面图

- 默认生成 `01-cover.png`
- 封面用于表达主题，不用于伪造证据
- 不得生成会让人误以为是真实产品截图、数据图或架构图的图片
- 使用generate-image技能生成封面时，提示词要明确要求“概念图”“卡通叙事风格”“暖色调”，并且符合主题，不要无关的装饰元素
- 封面图的宽高比是2.35：1

### 网页图片提取要求

- 用浏览器上下文判断图片是否和正文强相关，不要只看文件名
- 优先保留正文里的步骤图、结果图、架构图、图表
- 如果图片点开后才是高清版本，可以进一步打开或下载对应资源
- 对于纯装饰性插图、头像、按钮图标、社交分享图，通常不保留

### Markdown 规则

- 本地图片路径写为 `../../assets/{ID}/{filename}`
- 图片尽量放在对应内容附近，不要全部堆到文末
- 每张图补中文 `alt`
- 若需要分享图，可在 frontmatter 添加 `ogImage: "../../assets/{ID}/01-cover.png"`

## 8. 生成 Markdown 文件

文件路径：`src/data/blog/{ID}-{slug}.md`

### frontmatter 要求

- `pubDatetime` 不复用原文发布日期，默认使用**安全发布时间**：
  - 以系统当前时间为基准，转换为 `Asia/Shanghai`（`+08:00`）
  - 再减去 `20` 分钟后写入 frontmatter（避免发布后因未来时间被列表过滤）
- `pubDatetime` 必须是 ISO 8601 且带时区偏移（例如 `+08:00`）
- 禁止使用无时区格式（如 `2026-03-23T10:00:00`）和纯日期格式（如 `2026-03-23`）
- 仅当用户明确要求“定时发布”时才允许未来时间；否则一律使用安全发布时间

```yaml
---
pubDatetime: YYYY-MM-DDTHH:mm:ss+08:00
title: "中文标题"
description: "80-120 字中文摘要"
tags: ["Tag1", "Tag2"]
slug: "{slug}"
# ogImage: "../../assets/{ID}/01-cover.png"
source: "{original_url}"
---
```

## 9. 文末参考链接

保留一个 `## 参考` 小节，至少包含原文链接。

只保留正文中真正引用过、确实有价值的外部链接，控制在 10 条以内。

## 10. 最终检查

发布前逐项确认：

- [ ] 全文是自然中文，不像模板化 AI 输出
- [ ] 保留了原文的关键锚点，没有借题发挥
- [ ] 标题准确，小标题够短、够清楚
- [ ] 如果是教程型文章，步骤已经写清楚：前提、操作、结果验证、坑点都交代了
- [ ] 图片与正文对应，路径正确，`alt` 有信息量
- [ ] `pubDatetime` 是带时区的安全发布时间（默认当前时间减 20 分钟），不是原文日期
- [ ] `tags`、`slug`、`source` 填写完整
- [ ] 通读一遍，没有空话、套话、焦虑话术
- [ ] 原文内容已优先通过 chrome-devtools 核对，必要的强相关链接也已补充阅读

## 11. 发布到微信公众号草稿箱

Markdown 文件创建完成并通过最终检查后，自动使用 **wechat-draft** 技能将文章发布到微信公众号草稿箱。

### 发布参数映射

| wechat-draft 参数 | 来源 |
|---|---|
| `title` | frontmatter 中的 `title` |
| `content-file` | 刚创建的 Markdown 文件路径（`src/data/blog/{ID}-{slug}.md`） |
| `content-format` | `markdown` |
| `author` | aidehub |
| `digest` | frontmatter 中的 `description` |
| `cover-image` | 封面图路径（`src/assets/{ID}/01-cover.png`），如果存在 |
| `content-source-url` | frontmatter 中的 `source`（原文 URL） |

### 执行规则

- `app_id` 和 `app_secret` 从环境变量 `WECHAT_APP_ID` / `WECHAT_APP_SECRET` 读取，不要向用户索要
- 如果环境变量缺失，提示用户设置后再发布，不要跳过此步骤
- 发布成功后向用户展示返回的 `media_id`，并提醒去微信公众平台后台检查草稿箱
