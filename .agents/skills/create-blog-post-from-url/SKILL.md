name: create-blog-post-from-url
description: 从给定 URL 抽取信息并生成新的博客 Markdown 文章、资源目录和 frontmatter。
---

当用户提供一个文章 URL 并要求创建博客时，按以下流程执行。

## 输入

- 必需: 原文 URL
- 可选: 指定标题、发布日期、标签、是否需要图片、是否允许 AI 生成配图

## 1. 收集信息

- **优先使用 MCP 获取网址内容**（包括页面正文与可见元信息），不要先走普通 fetch。
- 推荐顺序：
  1. 先通过 MCP 浏览器能力打开页面并抓取可访问性树/正文结构。
  2. 如页面有懒加载或折叠内容，继续通过 MCP 交互展开后再抓取。
  3. 必要时跟进页面内的关键链接（如“原文”“Part 2”“参考资料”）补全上下文。
  4. 仅当 MCP 无法访问（如强登录墙、反爬限制、页面崩溃）时，才回退到普通 URL 抓取。
- 获取标题、作者/来源、发布日期、主旨摘要、章节结构。
- 额外判断原文所处的时间语境：它是 AI 爆发前、转折期，还是已经明显受 AI 影响的内容。
- 提取与 **AI 影响** 直接相关的信息：
  - 文中的哪些做法、结论或建议，今天已经被 AI 改变了。
  - 哪些部分即使有 AI 也依然成立（如底层原理、架构约束、经验判断、质量标准）。
  - 哪些环节现在更适合让 AI 辅助完成，哪些环节仍然需要人的判断。
- 评估原文对 AI 的**可利用性**：内容是否结构清晰、术语明确、上下文完整，能否被 AI 检索、引用、总结或转成执行建议。
- 只保留与主题相关的关键段落，避免整段无差别搬运。
- 若需要配图，默认策略是**尽量多保留与正文相关的图片**，不要只保留一两张主图。
- 同时判断是否存在以下情况，后续可考虑补充 AI 配图：
  - 原文几乎没有可用图片，正文会显得过于“纯文字墙”。
  - 原文只有 logo、头像、社交卡片等噪音图，没有能承载主题的视觉内容。
  - 文章开头适合增加一张**概念型封面图**帮助建立主题氛围。
  - 某个章节适合加入**概念型插图**来承接观点，但不适合伪造截图、架构图或数据图。
- 抓取图片时同时覆盖以下来源：
  - `<img src>`、`data-src`、`data-lazy-src`、`data-original`
  - `<picture><source srcset>` 与 `img[srcset]`
  - Markdown 图片语法 `![alt](url)`
  - CSS 内联背景图（仅当该图出现在正文核心区域）
- 对每张候选图片记录：原始 URL、所在段落/章节、原始 alt/title、尺寸信息（若可得）。
- 如果判断需要 AI 配图，同时记录每张计划生成图片的用途：封面图、章节插图、还是结尾概念图，并写明它要服务哪一段内容。

## 2. 翻译与写作

### 2.1 基本要求

- **必须将原文翻译为中文**，包括标题、描述和正文内容。
- 保持技术术语的准确性，专有名词保留英文，首次出现时括号附中文说明。
- 代码示例保持原样，注释翻译为中文。

### 2.2 写作风格（核心）

本博客作者的文风是**有观点的实践者对同行说话**，语气直接、自信、对话感强。生成内容必须严格匹配以下风格特征：

#### 开头：从问题或场景切入

- 用一个真实问题、当下事件或具体场景开场。
- **禁止**用"本文介绍..."、"随着...的发展"、"在当今..."这类空洞开头。

#### 段落：用连贯的叙述，不用碎片化列表

- 概念性内容写成**流畅的段落**，不要拆成项目符号列表。
- 只在并列条目（如配置项、命令列表、对比维度）时才用列表。
- 变换句子长度：短句制造冲击力，长句展开论述。
  - 短句："规律很明确。"、"我选择乐观。"
  - 长句自然展开，不刻意断句。

#### 结构：自然分节，不要机械编号

- 用 `##` / `###` 标题自然分节，**不要用"一、二、三"或"第一步、第二步"编号**。
- 标题用直接的问题或陈述："为什么规划很重要"、"实际效果"。
- 用 `>` 引用块提炼关键洞察或原则。
- 适当使用**对比表格**替代冗长的文字对比。

#### 技术内容：先说为什么，再给代码

- 先解释动机和背景，再展示技术方案。
- 代码示例要**完整、可运行、贴近真实场景**（用 `OrderStatus` 这类领域模型，不用 `foo/bar`）。
- 假定读者有中级水平，不解释标准库函数或基础概念。

#### 结尾：不要写"总结"段

- 用前瞻性陈述或简短有力的结论收尾。
- **禁止**写"总之"、"综上所述"、"本文介绍了..."这类总结段。

### 2.3 绝对禁止的 AI 痕迹

以下模式会让读者一眼看出是 AI 生成的，**必须完全避免**：

**禁止的结构词和过渡语：**

- "首先...其次...最后..."
- "值得注意的是"、"需要强调的是"、"需要指出的是"
- "让我们来看看"、"下面我将介绍"、"接下来让我们"
- "总之"、"综上所述"、"总而言之"
- "在...方面"、"从...角度来看"

**禁止的句式：**

- "不是...，而是..."
- "...的关键在于..."（偶尔一次可以，不能反复出现）
- "这意味着..."（同上）
- "...提供了一种...的方式"
- "值得一提的是"

**禁止的标点和格式：**

- 破折号（用逗号或分句替代）
- 感叹号连用
- 过度加粗（每段最多加粗一处关键词）

**禁止的词汇：**

- 互联网黑话：赋能、抓手、对齐、颗粒度、方法论、落地、心智、赛道、沉淀、闭环、链路
- 空洞修饰：极大地、显著地、无疑、毋庸置疑、不言而喻
- AI 高频词：探索、深入、全面、详细、强大的、优雅的

### 2.4 鼓励使用的写法

- **"这就是..."** 用来点明核心洞察
- **直接的反问或设问**作为段落或章节开头
- **括号补充**英文术语或简短说明，不打断主句
- **具体数字和真实案例**替代抽象描述
- **短句收尾**制造节奏感："规划会翻转这个局面。"
- **第一人称**自然穿插，体现实践者视角

### 2.5 加入 AI 时代的编辑视角

整理文章时，不能只做“翻译 + 摘要”，还要主动判断这篇内容在 AI 语境下是否仍然成立，以及应该如何被重新组织。

生成正文时，至少在行文中明确回答下面三个问题（可以自然融入章节，不必机械写成固定小标题）：

- **AI 已经改变了什么**：哪些流程、门槛、成本、学习路径，因为 AI 工具出现而发生了变化。
- **什么没有被改变**：哪些判断、原则、约束、工程权衡，依然不能外包给 AI。
- **读者现在应该关注什么**：如果今天重新实践这个主题，注意力应该放在哪些新的关键点上。

如果原文本身写于 AI 浪潮之前，要主动补上“放到今天看，它哪里过时了，哪里仍然值钱”。不要假装时间没有发生变化。

如果原文已经讨论 AI，也不要停留在表面描述。要继续追问：

- 这是“AI 能帮忙”还是“AI 改写了工作方式”？
- 它提升的是速度、质量，还是决策质量？
- 它带来了哪些新的风险，例如幻觉、上下文丢失、错误自信、维护成本上升？

对明显会被 AI 取代为“常识”的内容，写作时要压缩篇幅；把重点转移到以下更有长期价值的信息：

- 决策依据与边界条件
- 失败案例、误区与反模式
- 真实项目中的取舍
- AI 不擅长隐含推断时需要人补充的上下文

同时，让文章本身更容易被 AI 正确使用：

- 关键术语前后一致，不随意换称呼。
- 因果关系讲清楚，避免“结论有了、前提没了”。
- 对方法、流程、策略尽量写出适用前提与不适用场景。
- 需要比较时，优先用紧凑的对比表格，让人和 AI 都能快速提取差异。
- 给出可引用的判断句，而不是只有情绪化表达。

## 3. 生成 ID 与 Slug

- 在 `src/data/blog` 中读取已有文件名，解析 `{ID}-` 数字前缀并递增。
- 将 ID 保持为 3 位数字（例如 `095` -> `096`）。
- Slug 使用 kebab-case；如果标题为中文，转成简短英文/拼音短语并保持唯一。

## 4. 处理图片资源与 AI 配图

### 4.1 目标与目录

- 在 `src/assets/{ID}/` 下保存图片资源。
- 目标是：**在不引入明显噪音图的前提下，最大化保留正文相关图片**。
- 若原图不足以支撑阅读体验，可在合适位置补充 1 张封面图，必要时再补 1-2 张概念型插图。
- 只有在没有任何可用图片时才允许不创建目录。

### 4.2 图片来源优先级

- 默认优先级：**原文可用图片 > 原文远程链接回退 > AI 生成配图**。
- **封面图必须始终使用 `generate-image` skill 生成**，无论原文是否有图片。每篇文章都需要一张风格统一的 AI 封面图（见 4.6 节的风格规范）。
- 对于章节插图，只有在以下情况才使用 `generate-image` skill 补图：
  - 原文没有高质量配图，或现有图片与正文几乎无关。
  - 文章主题偏抽象（如方法论、趋势判断、AI 工作流、工程文化），适合用概念型插画做视觉承接。
- 不要为了"看起来丰富"而给每个小节都生成一张图。默认控制在：
  - 1 张封面图（**必须生成**）；
  - 0-2 张章节插图（视内容需要）；
  - 总数优先少而准。

### 4.3 保留与过滤规则

- 默认保留：架构图、流程图、代码截图、图表、步骤截图、示意图。
- 默认过滤：导航/页脚图标、社交 icon、头像、广告位、追踪像素（如 1x1）。
- 对“是否保留”拿不准时，**倾向保留**，避免遗漏关键图。
- 若图片尺寸可得，优先过滤极小图（例如宽 < 160 或高 < 90），但要避免误删数学公式小图。
- 即使打算补 AI 配图，也不要先丢掉能表达事实的原图。原图承担“证据”和“上下文”，AI 图只承担“气氛”和“引导”。

### 4.4 下载策略（优先保证成功率）

- 先将相对路径图片 URL 解析为绝对 URL。
- 对 `srcset` / `picture`，优先选清晰度更高的版本。
- 下载时带上合理请求头（尤其 Referer 指向原文页），降低 403 风险。
- 并行下载（建议并发 3-5），单张失败自动重试 1 次。
- 对于懒加载页面，先滚动或交互触发图片加载，再抓取最终 URL。
- 对 data URI（base64）图片，直接解码保存为本地文件。

### 4.5 文件命名与格式

- 保留原始格式（png/jpg/webp/gif/svg）。
- 文件名建议：`{序号}-{语义名}.{ext}`，例如 `01-clean-architecture-diagram.png`。
- 若原文件名本身有语义，可沿用并做 kebab-case 规范化。
- 若无法确定语义名，使用稳定命名：`{序号}-image.{ext}`。
- 对 AI 生成图片，优先使用稳定命名：
  - `01-cover.png`
  - `02-inline-{topic}.png`
  - `03-inline-{topic}.png`
- 如果原文已经有 `01-...` 图片编号，则继续顺延，不要覆盖已有文件。

### 4.6 使用 `generate-image` skill 生成配图

#### 封面图（必须生成）

每篇文章都**必须**调用 `generate-image` 生成一张封面图，使用以下固定风格规范：

**固定艺术风格**：温暖色调的卡通风格（warm color palette, cartoon style）
- 柔和的灯光与温馨的氛围
- 简洁而生动的卡通线条
- 固定画幅比例：**2.35:1（电影画幅）**
- 传达治愈、舒适或积极的情感

**封面图提示词生成流程**：
1. 仔细阅读文章的主题和核心关键词。
2. 提炼最能代表该内容的核心视觉元素（人物动作、场景氛围、关键物件）。
3. 将视觉元素转化为详细的英文绘图描述。
4. 提示词中**必须包含**：`warm color palette`, `cartoon style`, `2.35:1 aspect ratio`。
5. 确保画面传达出治愈、舒适或积极的情感，与"温暖"设定相符。

**封面图提示词模板**：
```
[文章核心场景描述，用英文描述人物、动作、场景氛围或关键物件], warm color palette, cartoon style, 2.35:1 aspect ratio, soft lighting, cozy atmosphere, clean vivid cartoon lines, no text, no watermark, no UI elements
```

例如，对于一篇关于"AI 辅助代码审查"的文章，提示词可以是：
```
A developer and a friendly robot collaborating at a glowing computer screen reviewing code together, warm color palette, cartoon style, 2.35:1 aspect ratio, soft lighting, cozy atmosphere, clean vivid cartoon lines, no text, no watermark
```

- 调用前先检查项目根目录 `.env` 是否已配置 `OPENROUTER_API_KEY`。若缺失，需要先补齐配置，再继续生成。

#### 章节插图（按需生成）

- 在以下位置可以额外调用 `generate-image`：
  - **大章节起始处的概念插图**：用于承接"AI 工作流""架构权衡""团队协作""工程文化"这类抽象主题。
- 章节插图同样遵循相同的风格规范：warm color palette, cartoon style, 2.35:1 aspect ratio。
- 主体要单一，避免塞太多细节导致文章视觉噪音。

#### 禁止使用 `generate-image` 伪造的内容

- 精确的产品 UI 截图
- 真实数据图表
- 架构图、流程图、时序图
- 会让读者误以为来自原文证据链的"事实型图片"

如果需要的是严谨技术图，而原文又没有可复用素材，宁可不生成，也不要拿概念图冒充事实图。

#### 生成后复核

生成后人工复核至少四件事：
- 画面是否与文章主题一致，风格是否符合温暖卡通风格。
- 是否错误暗示了不存在的产品界面、代码内容或统计结论。
- 是否包含可见文字、水印、奇怪手部、畸形元素或明显 AI 瑕疵。
- 是否已使用 2.35:1 横向构图（适合博客封面展示）。
### 4.7 Markdown 替换与插入规则

- 将正文中的远程图片引用替换为本地相对路径：
  `../../assets/{ID}/{filename}`
- 尽量保持图片在原文中的出现位置，不要集中移动到文末。
- 每张图片补全简洁、具体的中文 `alt`。
- 对 AI 生成封面图，默认放在 frontmatter 后、正文开头之前，格式参考：

```markdown
![文章标题或主题概念图](../../assets/{ID}/01-cover.png)
```

- 若该封面图同时适合作为社交分享图，可在 frontmatter 中增加：

```yaml
ogImage: "../../assets/{ID}/01-cover.png"
```

- 对 AI 生成章节插图，只放在它真正能帮助读者切换语境的位置，不要密集插入。
- 某张图下载失败时：
  - 保留原始远程链接，避免内容断裂；
  - 紧邻处添加 HTML 注释说明失败原因与原 URL，便于后续人工补抓。
- 某张 AI 图生成失败时：
  - 直接放弃该图或降级为仅保留原图/纯文字，不要为了补图而反复堆砌低质量结果；
  - 如有必要，在工作说明中记录“原计划生成封面图，但因效果不佳/生成失败而取消”。

### 4.8 特殊场景

- 防盗链/CDN 失败：优先重试并带 Referer；仍失败则保留远程链接并注释。
- 动图（GIF/WebP 动画）：优先保留原动画格式。
- SVG：优先保存原始 SVG，不转位图。
- 多页文章（Part 1/2/3）：按需跟进分页并合并抓取图片，避免漏图。
- 如果原文已有高质量封面或关键示意图，不要再额外生成“装饰性重复封面”。
- 如果文章核心价值来自截图步骤教学，优先保留步骤截图；AI 配图只能作为开头引导，不能替代步骤图。

## 5. 生成 Markdown 文件

- 文件路径: `src/data/blog/{ID}-{slug}.md`
- 使用以下 frontmatter 模板并填充**中文内容**:

```yaml
---
pubDatetime: YYYY-MM-DD
title: "文章标题（中文）"
description: "80-120 字中文摘要，概括文章核心观点"
tags: ["Tag1", "Tag2"]
slug: "{slug}"
# 若有本地封面图且适合作为分享图，可补充：
# ogImage: "../../assets/{ID}/01-cover.png"
source: "{original_url}"
---
```

- **不要使用固定的章节模板**（如"背景/关键要点/实践建议"）。根据原文内容自然组织结构，每篇文章的章节应该不同。
- 好的章节标题参考："问题出在哪"、"换个思路"、"实际跑起来是什么样"、"什么时候该用"。
- 如果主题明显受到 AI 影响，正文中应自然加入以下维度中的一个或多个：
  - 今天这件事被 AI 改写到了什么程度
  - 哪些内容可以交给 AI，哪些不能
  - 对读者来说，现在最值得补的不是知识点，而是判断力、上下文还是验证能力
- 不要求每篇都单独写一个名为“AI 影响”的章节，但最终文章必须让读者看得出：作者已经思考过 AI 如何改变这件事。

## 6. 文末参考链接

在文章正文结尾处添加参考链接区块，格式如下：

```markdown
## 参考

- [原文](原始URL) — 作者/来源（若可得）
- [相关链接标题](URL) — 简短说明（可选）
```

规则：

- **原文链接必须包含**，使用原始 URL，显示文本为"原文"加文章标题（如"原文: Building Clean APIs"）或直接写"原文"。
- 若正文中引用了外部资料（如官方文档、GitHub 仓库、相关博文），一并列入参考区块。
- 行文中若已有内联超链接，可将其汇总至此处，避免正文链接列表化，同时保持内联链接不变。
- 不超过 10 条，优先列出对读者最有价值的链接，去掉广告页、社交主页等低价值 URL。

## 7. 质量检查

- **确认所有内容均为中文**，技术术语翻译准确。
- **逐段检查 AI 痕迹**：对照 2.3 节的禁止清单，逐一排查并改写。
- 段落是否像人在说话？读出来是否自然？如果像机器翻译或 AI 生成，重写该段。
- **检查 AI 时代的有效性**：
  - 文章有没有交代这篇内容在今天是否仍然成立。
  - 如果 AI 已改变该主题，文中是否写清“变了什么、没变什么、现在该关注什么”。
  - 是否避免把已经被 AI 商品化的常识写成全文重点。
- **检查内容是否便于 AI 使用**：
  - 关键概念是否定义清楚、命名一致。
  - 结论是否能追溯到前提、案例或对比。
  - 是否写出了适用边界，而不是只给抽象正确的话。
- `tags` 保持 2-6 个，包含技术主题（如 `.NET`, `Azure`, `Architecture`）。
- `source` 必须指向原始 URL。
- 文章中图片路径与目录保持一致。
- 图片检查清单：
  - `src/assets/{ID}/` 中确实存在被引用的图片文件。
  - Markdown 中优先使用本地路径，只有失败项保留远程 URL。
  - 图片顺序与正文语义一致，不出现“图文错位”。
  - `alt` 文本为中文且有信息量，不使用空值或 `image` 这类占位词。
  - 对失败下载项已写明注释（便于后续人工修复）。
  - AI 生成图片只承担概念表达，不冒充原文证据、截图或图表。
  - 如果设置了 `ogImage`，它必须指向实际存在的本地图片文件。
- **最终自测**：把文章前三段放到"AI 检测器"的视角审视，如果能被识别为 AI 生成，回去改写。
