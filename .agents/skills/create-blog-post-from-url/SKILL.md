当用户提供一个文章 URL 并要求创建博客时，按以下流程执行。

## 输入

- 必需: 原文 URL
- 可选: 标题、原文发布日期、标签

`原文发布日期` 只用于理解上下文、判断时效性和正文表述，不用于生成 frontmatter 里的 `pubDatetime`。

## 1. 收集信息

**抓取优先级**：MCP 浏览器 > MCP 交互展开懒加载/折叠 > 跟进关键链接（原文/Part 2/参考资料） > 普通 URL 抓取（仅 MCP 不可用时）。

**提取内容**：标题、作者/来源、发布日期、主旨摘要、章节结构。只保留主题相关段落，不整段搬运。

**AI 时代语境判断**（后续写作需要）：
- 哪些做法/结论已被 AI 改变
- 哪些依然成立（底层原理、架构约束、经验判断）
- 哪些环节适合 AI 辅助，哪些需要人的判断
- 内容是否结构清晰、术语明确，能被 AI 检索引用

**图片抓取**：
- 来源覆盖：`<img src/data-src/data-lazy-src/data-original>`、`<picture><source srcset>`、`img[srcset]`、Markdown `![]()`、核心区域 CSS 背景图
- 每张记录：原始 URL、所在章节、alt/title、尺寸
- 默认策略：仅保留必要的正文相关图片
- 评估是否需要补充 AI 配图（纯文字墙、只有噪音图、适合概念封面/插图）

## 2. 翻译与写作

### 2.0 先判定文章类型

在动笔前，先判断原文更接近哪一类，再决定写法。不要所有文章都套成同一种 AideHub 腔调。

常见类型：
- **教程 / 实操型**：重点是问题、方案、步骤、注意事项
- **产品更新 / 发布型**：重点是更新点、意义、适用人群、值不值得关注
- **观点评论型**：重点是论点拆解、现实意义、判断边界，不要写成教程
- **研究 / 论文解读型**：重点是核心发现、方法限制、现实可迁移性
- **案例复盘型**：重点是背景、决策、过程、教训
- **工具评测型**：重点是体验、优缺点、适用场景、与替代方案的差异

写之前先回答三件事：
1. 这篇的核心价值是新信息、作者观点，还是操作方法？
2. 读者读完最该带走的是“会做一件事”，还是“形成一个判断”？
3. 这篇应该更像教程、评论、新闻解读，还是实验记录？

后续结构、段落密度、图片策略、结尾方式都要服从这个判断。

### 2.1 基本要求

- **必须翻译为中文**（标题、描述、正文），代码注释也翻译
- 专有名词保留英文，首次出现括号附中文

### 2.2 写作风格

文风：**有观点的实践者对同行说话**，直接、自信、对话感强。

### 2.2.0 保留原文锚点，避免借题发挥

每篇至少保留 **2-4 个来自原文的具体锚点**，让文章明显建立在原文之上，而不是只借原文做引子。

锚点可以是：
- 原作者的关键分类
- 一个具体实验现象
- 一个代表性例子
- 一个特别有辨识度的判断
- 一个数据点、术语或原始场景

要求：
- 可以转述，不必硬引用原句
- 但不能把原文所有具体性都洗成抽象总结
- 如果删掉原文链接后，整篇更像你自己借题发挥，而不像在解读原文，说明改写过头了

对于**观点评论型**原文尤其注意：优先保留原作者的判断框架，不要把原作者文章改写成 AideHub 自己本来想说的话。

写这类文章的目的，不是制造焦虑、渲染“再不跟上就晚了”，而是：
- 解释清楚一个工具、技术或方法到底在解决什么问题
- 帮读者快速理解它能带来什么便利、节省什么成本、改善什么流程
- 把事情讲清楚，方便以后被人或 AI 检索、引用、回查

| 维度 | 要求 |
|------|------|
| 开头 | 从真实问题/当下事件/具体场景切入。禁止"本文介绍..."、"随着...的发展"、"在当今..." |
| 段落 | 概念性内容写连贯段落，不拆成列表。只在并列条目时用列表。长短句交替 |
| 结构 | `##`/`###` 自然分节，标题用问题或陈述。不用"一、二、三"编号。用 `>` 引用块提炼洞察 |
| 技术 | 先说动机再给代码。代码完整可运行、用真实领域模型（非 foo/bar）。假定中级读者 |
| 结尾 | 根据文章内容自然收尾，可以是适用边界、使用建议、实际价值、后续观察点。禁止每篇都用同一种“压缩成一句话”模板收尾。禁止"总之"、"综上所述"、"本文介绍了..." |

### 2.2.2 控制观点密度，留出呼吸感

尤其是**观点评论型**和**趋势解读型**文章，不要段段都写成结论句。即使每段都“说得对”，连在一起也容易变得紧绷，像高质量但过密的 AI 输出。

要求：
- 每 3-5 段里，至少允许有 1 段以解释、铺垫、例子或场景推进，而不是直接下判断
- 避免连续几段都用“这很关键”“真正的问题是”“这也是为什么”这种强调句起手
- 如果一段删掉 1-2 个判断句后更顺，那就应该删
- 对观点文，宁可保留节奏变化，也不要追求句句都像可摘录金句

### 2.2.1 标题规则（重点）

标题必须优先做到：**准确概括原文主题**，其次才是表达角度。

- **默认写法**：用朴素、直白、能概括内容的中文陈述句
- **优先保留文章对象**：技术名词、产品名、方法名、主题词要尽量留在标题里
- **可以有观点，但不能抢戏**：观点只能帮助理解主题，不能盖过主题本身
- **如果拿不准，宁可平实，不要耍花样**

**禁止标题党和过度包装：**
- 不要故意制造反差、悬念、冲突感
- 不要把普通经验写成“颠覆”“真相”“本质”“终极答案”
- 不要为了显得有判断就硬写成长句、双重转折句
- 不要把文章明明在讲 A，标题却写成更大的 B

**尽量避免的标题句型：**
- “当 X 之后，真正重要的是 Y”
- “X 之外，更关键的是 Y”
- “X 不只是 A，而是 B”
- “你以为是 A，其实是 B”
- “为什么真正该关注的是 Y”

这些句型不是绝对禁用，但默认不用。只有当原文核心论点本来就是这种结构，而且不用它就会失真时，才允许使用。

**推荐标题策略：**
1. 先写一个“如果发在文档站也成立”的标题
2. 再检查它是否准确反映原文主题、范围、对象
3. 最后才决定要不要轻微加入判断

**推荐风格示例：**
- “pnpm 如何用 Git Worktree 支持多 Agent 并行开发”
- “GitHub Copilot CLI 的 Slash Commands 有什么用”
- “OpenAI 如何看待 Harness Engineering”
- “为什么有些 Agent 工作流更适合用 SKILL.md 而不是 MCP”

**不推荐风格示例：**
- “当终端里的 AI 不再只靠聊天，真正重要的是 Slash Commands”
- “多个 Agent 同时开发时，先扛不住的其实不是 Git”
- “代码不再稀缺之后，工程团队真正该做什么”

判断标准：读完标题，用户应该立刻知道“这篇在讲什么”，而不是先感受到“这标题挺会说”。

### 2.3 禁止的 AI 痕迹

**禁止的过渡语**："首先...其次...最后..."、"值得注意/强调/指出的是"、"让我们来看看"、"下面我将介绍"、"总之/综上"、"在...方面"、"从...角度来看"

**禁止的句式**："不是...而是..."、"...的关键在于..."（偶尔一次可）、"...提供了一种...的方式"、"值得一提的是"

**禁止的格式**：破折号（用逗号或分句）、感叹号连用、过度加粗（每段最多一处）

**禁止的词汇**：
- 互联网黑话：赋能、抓手、对齐、颗粒度、方法论、落地、心智、赛道、沉淀、闭环、链路
- 空洞修饰：极大地、显著地、无疑、毋庸置疑、不言而喻
- AI 高频词：探索、深入、全面、详细、强大的、优雅的

**鼓励**："这就是..."点明洞察、设问开头、括号补充术语、具体数字和案例、短句收尾、第一人称穿插

### 2.4 AI 时代编辑视角

正文中可以自然融入以下三个维度（不必每篇都硬凑齐，更不要写成模板化口号）：
1. **AI 已改变什么**：流程、门槛、成本、学习路径的变化
2. **什么没变**：判断、原则、约束、工程权衡
3. **读者现在该关注什么**：新的关键实践点

对 AI 浪潮前的原文，主动补上"今天看哪里过时、哪里仍有价值"。对已讨论 AI 的原文，追问：是"AI 能帮忙"还是"AI 改写了工作方式"？提升的是速度、质量还是决策质量？带来了哪些新风险（幻觉、上下文丢失、错误自信）？

不要为了显得“有时代感”就强行放大焦虑、危机感或替代叙事。重点仍然是讲清楚：这项技术/工具具体提供了什么便利、改善了什么流程、适合什么场景、不适合什么场景。

对已成 AI 常识的内容压缩篇幅，重点放在：决策依据与边界条件、失败案例与反模式、真实取舍、AI 不擅长时需要人补充的上下文。

让文章便于 AI 使用：术语前后一致、因果关系完整、写出适用前提与不适用场景、用对比表格替代冗长比较、给出可引用的判断句。

## 3. 生成 ID 与 Slug

- 读取 `src/data/blog` 已有文件，解析数字前缀并递增，保持 3 位（如 `095` → `096`）
- Slug 用 kebab-case；中文标题转简短英文/拼音短语，保持唯一

## 4. 图片处理

### 4.1 目录与目标

图片保存到 `src/assets/{ID}/`。最大化保留正文相关图片，无噪音图。无任何可用图片时才不创建目录。

### 4.2 来源优先级

- **原文图片 > 远程链接回退 > AI 生成**
- **封面图必须用 `generate-image` 生成**（每篇都要，见 4.5）
- 章节插图仅在原文缺图或主题偏抽象时补充
- 控制总量：1 张封面图（必须）+ 0-2 张章节插图

### 4.2.1 按文章类型决定配图策略

不要对所有文章一律追求“有封面、有配图”。先看文章类型：

- **观点评论型**：正文可少图，重点靠结构和判断推进；封面偏隐喻型，正文宁缺毋滥
- **实验 / 评测型**：优先保留原文输入图、结果图、对比图；这是“证据”
- **教程 / 实操型**：优先步骤图、界面图、代码图；AI 图只做开头引导
- **产品更新 / 发布型**：优先产品图、官方示意图、功能截图
- **架构 / 流程型**：优先架构图、流程图、对比图；封面再做抽象隐喻

判断标准：正文图片的作用应该是**增加信息量或建立证据感**，不是单纯打断文字墙。

### 4.3 保留与过滤

- 保留：架构图、流程图、代码截图、图表、步骤截图、示意图
- 过滤：导航/页脚图标、社交 icon、头像、广告位、追踪像素（< 160×90）
- 拿不准时倾向保留。原图承担"证据"，AI 图只承担"气氛"

### 4.4 下载与命名

- 相对路径解析为绝对 URL；`srcset`/`picture` 优选高清版；带 Referer 请求头；并发 3-5，失败重试 1 次
- 懒加载页面先交互触发；base64 直接解码保存
- 命名：`{序号}-{语义名}.{ext}`（如 `01-clean-architecture-diagram.png`），无语义名用 `{序号}-image.{ext}`
- AI 生成图：`01-cover.png`、`02-inline-{topic}.png`
- 保留原始格式（png/jpg/webp/gif/svg），已有编号则顺延

### 4.5 AI 配图（`generate-image` skill）

#### 封面图（每篇必须生成）

**风格目标**：保持**偏卡通、略抽象、适合技术文章**的统一基调，但**绝不能千篇一律**。封面必须根据文章主题决定主体、构图、色调和氛围。

**长期偏好**：
- 默认偏好温暖色调，但**允许按主题切换色温**
- AI / agent / workflow / 编程协作：暖橙、金黄、珊瑚、柔和蓝紫，可做“温暖科技感”
- 安全 / 风险 / 审计 / 故障：可转深蓝、冷灰、暗红点缀，保留克制感
- 数据 / 架构 / 抽象概念：可用更清爽的青蓝、米白、低饱和撞色
- 效率工具 / 教程 / 产品更新：更明亮、轻快、清晰

**不要再固定一套 prompt 直接复用。** 每次都先判断：
1. 文章属于哪类主题（工程、工具、架构、文化、风险、数据、工作流）
2. 最适合的主体是什么（人物协作 / 抽象装置 / 流程隐喻 / 文档 / 代码空间 / 乐谱 / 图纸等）
3. 色调应该偏暖、暖中带冷，还是局部冷色压气氛
4. 画面更适合“温馨协作”“理性分析”“紧张风险”还是“轻巧实验感”

**硬性保留元素**：
- `cartoon style`
- `2.35:1 aspect ratio`
- `clean vivid cartoon lines`
- `no text, no watermark, no UI elements`

**推荐可变元素**（按主题替换，不必全写死）：
- 色调：`warm color palette` / `muted warm palette` / `warm-cool contrast` / `deep blue with warm accents`
- 光线：`soft lighting` / `dramatic lighting` / `gentle ambient glow`
- 氛围：`cozy atmosphere` / `focused analytical mood` / `subtle tension` / `playful experimental feel`
- 风格补充：`slightly abstract` / `editorial illustration` / `minimal visual metaphor`

**提示词流程**：提炼文章核心视觉隐喻 → 决定画面主体与情绪 → 决定色调方案 → 转成英文绘图描述 → 再补固定限制词。

**模板**：
```
[核心场景英文描述], [与主题匹配的色调描述], cartoon style, 2.35:1 aspect ratio, [与主题匹配的光线/氛围描述], clean vivid cartoon lines, slightly abstract, no text, no watermark, no UI elements
```

**示例 1（AI 辅助代码审查）**：
```
A developer and a friendly robot collaborating at a glowing computer screen reviewing code together, warm color palette, cartoon style, 2.35:1 aspect ratio, soft lighting, cozy atmosphere, clean vivid cartoon lines, slightly abstract, no text, no watermark, no UI elements
```

**示例 2（安全 / 风险主题）**：
```
A software system under inspection with warning signals, audit trails, and a cautious engineer observing hidden weak points, deep blue tones with warm red accents, cartoon style, 2.35:1 aspect ratio, focused analytical mood, clean vivid cartoon lines, slightly abstract, no text, no watermark, no UI elements
```

**示例 3（抽象架构 / 工作流主题）**：
```
Interconnected modules, documents, and agents moving through a structured engineering workflow, warm-cool contrast palette, cartoon style, 2.35:1 aspect ratio, gentle ambient glow, minimal visual metaphor, clean vivid cartoon lines, no text, no watermark, no UI elements
```

调用前检查 `.env` 中 `OPENROUTER_API_KEY` 是否配置。

#### 章节插图（按需）

延续文章封面的主题语言，但允许比封面更功能化。用于抽象主题（AI 工作流、架构权衡、工程文化）的大章节起始处，或用于补足原文缺少关键对比图时的视觉节奏。主体单一，避免视觉噪音；章节图和封面不要只是同一张图换个姿势。

#### 禁止伪造

不得生成：产品 UI 截图、数据图表、架构/流程/时序图、任何会被误认为原文证据的图片。

#### 生成后复核

- 画面与文章主题一致，符合温暖卡通风格
- 无错误暗示的界面/代码/统计结论
- 无可见文字、水印、畸形元素
- 已使用 2.35:1 横向构图

### 4.6 Markdown 图片规则

- 远程图片替换为本地路径：`../../assets/{ID}/{filename}`
- 保持图片在原文位置，不集中到文末
- 每张图补中文 `alt`（有信息量，非空值/`image`）
- 封面图放在 frontmatter 后、正文前：`![主题概念图](../../assets/{ID}/01-cover.png)`
- 适合作分享图时 frontmatter 加 `ogImage: "../../assets/{ID}/01-cover.png"`
- 下载失败：保留远程链接 + HTML 注释说明原因
- AI 图生成失败：直接放弃，不堆砌低质量结果

### 4.7 特殊场景

- 防盗链失败：重试带 Referer，仍失败保留远程链接并注释
- 动图保留原格式，SVG 保留原始 SVG
- 多页文章跟进分页合并抓取
- 原文已有高质量封面时不生成重复装饰封面
- 步骤教学类文章优先保留步骤截图，AI 配图仅作开头引导

## 5. 生成 Markdown 文件

文件路径：`src/data/blog/{ID}-{slug}.md`

### 5.1 `pubDatetime` 规则

- `pubDatetime` **必须使用生成文章当下的系统当前日期时间**
- **不要**复用原文发布日期，也不要只写 `YYYY-MM-DD`
- 生成 frontmatter 前，先读取当前系统时间
- 可直接使用命令获取时间，例如 PowerShell: `Get-Date -Format "yyyy-MM-ddTHH:mm:sszzz"`，POSIX shell: `date -Iseconds`
- 使用 **ISO 8601** 且带时间部分，优先包含时区偏移
- 示例：`2026-03-12T12:01:23+08:00`

```yaml
---
pubDatetime: YYYY-MM-DDTHH:mm:ss+08:00
title: "中文标题"
description: "80-120 字中文摘要"
tags: ["Tag1", "Tag2"]
slug: "{slug}"
# ogImage: "../../assets/{ID}/01-cover.png"
source: "{original_url}"
---
```

- **不用固定章节模板**，根据内容自然组织，每篇结构不同
- 标题先求准确概括主题，再考虑是否加入轻微判断；默认使用朴素陈述句
- 标题检查：去掉修辞后，是否还能一眼看出文章对象、范围、核心主题？如果不能，重写
- AI 相关主题须在正文中体现：被改写程度、哪些能/不能交给 AI、读者该补判断力还是验证能力

- 不超过 10 条，去掉广告/社交主页等低价值 URL

## 7. 质量检查

- [ ] 所有内容为中文，技术术语准确
- [ ] 已先判定文章类型，写法与类型匹配，没有把评论写成教程、把更新写成泛议论文
- [ ] 至少保留 2-4 个原文具体锚点，读者能看出这篇确实建立在原文之上
- [ ] 逐段对照 2.3 节禁止清单，排查 AI 痕迹；读出来不自然就重写
- [ ] 没有为了制造焦虑而夸大趋势、风险或替代感
- [ ] 文章清楚说明了：这项技术/工具/做法具体带来什么便利，适合什么场景
- [ ] 观点密度不过载：不是段段都在高强度下判断，至少有部分段落承担解释、铺垫、举例功能
- [ ] 结尾是顺着正文自然收住的，不是套同一个“压缩成一句话”模板
- [ ] AI 时代有效性：如确有必要，再交代"变了什么、没变什么、该关注什么"
- [ ] 内容便于 AI 使用：术语一致、结论可溯源、有适用边界
- [ ] `tags` 2-6 个，含技术主题；`source` 指向原始 URL
- [ ] `pubDatetime` 是生成时的当前日期时间，包含时间部分，不是原文发布日期
- [ ] 图片策略与文章类型一致，正文图片承担信息量或证据感，而不是纯装饰
- [ ] 图片路径与 `src/assets/{ID}/` 实际文件一致
- [ ] `alt` 为中文且有信息量
- [ ] 下载失败项有注释；AI 图仅概念表达，不冒充证据
- [ ] `ogImage` 指向实际存在的本地文件
- [ ] 前三段放到"AI 检测器"视角审视，能被识别就改写
- [ ] 最后再以“站点主编”视角通读一遍：这篇像不像 AideHub 会发的文章？有没有哪段虽然正确但显得像 AI 最容易写出来的正确废话？
